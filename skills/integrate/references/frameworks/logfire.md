## 2. Minimum-change tier

`bootstrap-file`

Logfire is OTel-native and installs its own `TracerProvider` inside
`logfire.configure()` via `trace.set_tracer_provider(...)`.

*Python.* The OTel API enforces a set-once latch on the global
provider, so if kubit-otel registers first, Logfire's registration
silently becomes a no-op and Logfire spans are lost. The Python
wiring in §3 therefore must run **after** `logfire.configure()` —
either via the merge form (`additional_span_processors=[...]`,
vendor-documented, preferred) or via the standalone form, which
calls `attach()` and fails loudly if invoked before Logfire
initializes.

*TypeScript (OTel JS SDK v2).* There is no post-hoc attach on Node —
v2 removed `addSpanProcessor` from `BasicTracerProvider` /
`NodeTracerProvider`, and `@kubit-ai/otel` does not export
`attach()`. The only supported Node pattern is to add
`new KubitSpanProcessor({ apiKey })` to the `spanProcessors: [...]`
array at the Logfire/OTel provider construction site. The
`@pydantic/logfire-api` browser/Edge surface does not own a
`TracerProvider` it exposes to user code; if the repo runs its own
`NodeSDK` / `NodeTracerProvider` alongside Logfire, that
construction site is the merge target.

## 3. Bootstrap snippet

Reference Kubit wiring — the *minimum* code that must end up in the
program (import + processor/`configure` call, plus any assertions
below).

Logfire's canonical extension API is the `additional_span_processors=`
kwarg on [`logfire.configure()`](https://logfire.pydantic.dev/docs/reference/api/logfire/).
Use the **merge form** whenever a `logfire.configure(...)` call is
found per §3a — it's one call-site edit and avoids a second file.
The `processors=` kwarg is deprecated and is **not** a drop-in for
`additional_span_processors=`; if the user is pinned to an older
Logfire, leave `processors=` alone and use the standalone form
instead. Use the **standalone form** when §3a finds no wiring site.

### Merge form — extend the existing `logfire.configure(...)` call

Python:

```python
# Add to the existing logfire.configure(...) call in the user's
# code — do not introduce a second call.
from kubit_otel import KubitSpanProcessor

logfire.configure(
    # …existing kwargs stay in place…,
    additional_span_processors=[
        KubitSpanProcessor(
            api_key=os.environ["KUBIT_EXPORT_API_KEY"],
            token_endpoint=os.environ["KUBIT_EXPORT_ENDPOINT"],
        ),
    ],
)
```

TypeScript: `@pydantic/logfire-api` does not expose an equivalent
kwarg, and OTel JS SDK v2 does not allow adding processors to an
already-running provider. The supported merge path is to construct
**one** `NodeTracerProvider` (or `NodeSDK`) with both the Logfire
processor/exporter and `KubitSpanProcessor` in its
`spanProcessors: [...]` at construction time:

```typescript
import { NodeSDK } from "@opentelemetry/sdk-node";
import { KubitSpanProcessor } from "@kubit-ai/otel";
// …plus the Logfire-side processor/exporter the host file already uses.

const sdk = new NodeSDK({
  spanProcessors: [
    // …existing Logfire processor stays first…,
    new KubitSpanProcessor({ apiKey: process.env.KUBIT_EXPORT_API_KEY! }),
  ],
});
sdk.start();
```

### Standalone form — `kubit_instrumentation.py` / `kubit-instrumentation.ts`

Python standalone: used when §3a finds no `logfire.configure(...)`
site to extend. Attaches on whatever provider Logfire registered
globally; must be imported AFTER Logfire initialization. `attach()`
raises if no real `TracerProvider` is registered yet — so misordering
fails loudly rather than silently creating a second provider that
would clobber Logfire's registration.

```python
# Generated by /kubit-integrate for logfire on <YYYY-MM-DD>.
# Must be imported AFTER logfire.configure().
# Requires KUBIT_EXPORT_API_KEY and KUBIT_EXPORT_ENDPOINT env vars.
import os
from kubit_otel import attach

attach(
    api_key=os.environ["KUBIT_EXPORT_API_KEY"],
    token_endpoint=os.environ["KUBIT_EXPORT_ENDPOINT"],
)
```

TypeScript standalone: OTel JS SDK v2 has no post-hoc attach, and
`@kubit-ai/otel` does not export `attach()`. If the repo uses
Logfire's Node/Edge API without a `NodeSDK` / `NodeTracerProvider`
construction site there is no merge target available; the skill
should in that case print the merge-form snippet above as a
reference and instruct the user to stand up a `NodeSDK` that hosts
both Logfire's and Kubit's processors in one `spanProcessors`
array. The skill must not emit a standalone file that calls
`configure({ ... })`, because that would register a second,
parallel `NodeTracerProvider` and clobber Logfire's registration.

## 3a. Integration-site signals

Grep targets for an existing wiring site to merge Kubit into. If
exactly one file in the repo matches, merge §3 wiring into that file;
otherwise fall back to the standalone bootstrap file per SKILL.md
step 9.

Python:

- A call to `logfire.configure(` — pass `KubitSpanProcessor(...)` into
  its `additional_span_processors=[...]` kwarg (vendor-documented,
  single call-site edit, no second file).

TypeScript:

- A `new NodeSDK({ spanProcessors: [...] })` or
  `new NodeTracerProvider({ spanProcessors: [...] })` construction in
  a module that also imports from `@pydantic/logfire-api` — add
  `new KubitSpanProcessor({ apiKey: process.env.KUBIT_EXPORT_API_KEY! })`
  to the same `spanProcessors` array at that construction site.
- If the repo uses Logfire's Node/Edge API but never constructs its
  own `NodeSDK` / `NodeTracerProvider`, there is no v2-compatible
  merge target; the skill prints the §3 merge-form snippet as a
  reference and asks the user to stand up a `NodeSDK` that hosts
  both processors.

If multiple files match, ask the user which one the agent's traces
flow through.

## 4. Wire-in instruction

Python: `{{KUBIT_IMPORT_STATEMENT}}` **after** `logfire.configure(...)`
in the same module. Example:

```python
import logfire
logfire.configure()
{{KUBIT_IMPORT_STATEMENT}}  # noqa: E402 — must follow logfire.configure()
```

TypeScript: the merge form edits an existing `NodeSDK` /
`NodeTracerProvider` construction site in place — no separate
bootstrap file is written, and the entrypoint already imports that
site, so no new import is required.

Required deps:

- Python: `pip install kubit-otel`
- TypeScript: `npm install @kubit-ai/otel` (requires
  `@opentelemetry/sdk-trace-base >= 2.0.0` as a peer — pin
  `@opentelemetry/sdk-node` / `sdk-trace-base` / `sdk-trace-node` /
  `resources` to the same `^2.x` major the project already uses).

Python:

```bash
KUBIT_EXPORT_API_KEY=<your-key> KUBIT_EXPORT_ENDPOINT=<your-endpoint> python -c "
{{KUBIT_IMPORT_STATEMENT}}
from opentelemetry import trace
trace.get_tracer('kubit-verify').start_span('hello-kubit').end()
```

```bash

const { trace } = require('@opentelemetry/api');
trace.getTracer('kubit-verify').startSpan('hello-kubit').end();

```
