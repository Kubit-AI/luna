# OpenAI Agents SDK Adapter (instrument)

## 1. Dependency signals

**Python only.** This adapter covers the OpenAI Agents SDK via the
`opentelemetry-instrumentation-openai-agents-v2` contrib bridge, which
is Python-only at the time of writing. The `@openai/agents` JS SDK has
no equivalent first-party OTel bridge yet, so TS repos are intentionally
not detected here — they should fall through to the `otel-genai` adapter
once the user has wired a manual `TracerProvider`.

- `openai-agents` in `pyproject.toml` / `requirements.txt`
- `from agents import Agent` / `from agents import function_tool`

Also flag as present-but-already-wired if the repo has:
- `opentelemetry-instrumentation-openai-agents-v2` in Python manifests
- any file that calls `OpenAIAgentsInstrumentor().instrument(`

In that case, the user already has an instrumentor installed; the skill's
job is only to add Kubit as a destination on the same tracer provider.

## 2. Minimum-change tier

`bootstrap-file`

The official OTel contrib package bridges the OpenAI Agents runtime into
OTel spans. We attach Kubit as an additional exporter on the provider
passed to that instrumentor.

## 3. Bootstrap snippet

Reference Kubit wiring — the *minimum* code that must end up in the
program (import + processor/`configure` call, plus any assertions
below). When merging into an existing wiring site (see §3a), adapt
placement and syntactic style to the surrounding file. The standalone
snippet is used verbatim only when no site is found.

**Gotcha — two separate tracing worlds.** The `@openai/agents` /
`openai-agents` SDKs ship their own tracing layer with
`add_trace_processor(...)` / `set_trace_processors(...)` APIs. Those
are **not** OTel `SpanProcessor`s and Kubit cannot attach to them
directly. The bridge below
([`opentelemetry-instrumentation-openai-agents-v2`](https://github.com/open-telemetry/opentelemetry-python-contrib/tree/main/instrumentation-genai/opentelemetry-instrumentation-openai-agents-v2))
translates Agents traces into OTel spans against a user-owned
`TracerProvider`, and Kubit attaches as a standard OTel
`SpanProcessor` on that provider. Existing native Agents trace
processors (e.g. Arize or Weights & Biases integrations plugged in
with `add_trace_processor(...)`) keep working in parallel — Kubit
does not interfere with them.

### Python — `kubit_instrumentation.py`

```python
# Generated by /kubit-integrate for openai-agents on <YYYY-MM-DD>.
# Requires KUBIT_EXPORT_API_KEY and KUBIT_EXPORT_ENDPOINT env vars.
# Requires: pip install opentelemetry-instrumentation-openai-agents-v2 kubit-otel
import os
from opentelemetry.instrumentation.openai_agents import OpenAIAgentsInstrumentor
from kubit_otel import configure

provider = configure(
    api_key=os.environ["KUBIT_EXPORT_API_KEY"],
    service_name="<service-name>",
    service_version="<service-version>",
    resource_attributes={
        "deployment.environment": "dev",
    },
    token_endpoint=os.environ["KUBIT_EXPORT_ENDPOINT"],
)
OpenAIAgentsInstrumentor().instrument(tracer_provider=provider)
```

### TypeScript

Not applicable — this adapter does not match TS repos (see §1). TS
users of `@openai/agents` should wire a manual `TracerProvider` and
re-run `/kubit-integrate`, which will then match the `otel-genai`

## 3a. Integration-site signals

Grep targets for an existing wiring site to merge Kubit into. If
exactly one file in the repo matches, merge §3 wiring into that file;
otherwise fall back to the standalone bootstrap file per SKILL.md
step 9.

Python:

- A call to `OpenAIAgentsInstrumentor().instrument(` — the
  `tracer_provider=` kwarg there is the merge point. Kubit's
  `configure(api_key=...)` returns a provider, which should be passed
  into that call (or, if a provider is already built above the
  instrumentor call, attach `KubitSpanProcessor` to it).
- A module that calls `add_trace_processor(` from `agents` alongside
  `TracerProvider(` construction.

TypeScript:

- Not applicable — this adapter does not match TS repos (see §1). TS
  users of `@openai/agents` who have already wired a manual provider
  will be matched by the `otel-genai` adapter instead.

If multiple files match, ask the user which one the agent's traces
flow through.
