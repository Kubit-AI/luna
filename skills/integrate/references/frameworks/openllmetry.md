# OpenLLMetry / Traceloop Adapter (instrument)

## 1. Dependency signals

- `traceloop-sdk` in `pyproject.toml` / `requirements.txt`
- `@traceloop/node-server-sdk` in `package.json`
- `from traceloop.sdk import Traceloop` / `from traceloop.sdk.decorators import`
- `import * as traceloop from "@traceloop/node-server-sdk"`
- `TRACELOOP_API_KEY`, `TRACELOOP_BASE_URL` env-var references
- `Traceloop.init(` call in Python or `traceloop.initialize(` in TS

## 2. Minimum-change tier

`bootstrap-file`

Traceloop owns the `TracerProvider` — `Traceloop.init(...)` (or
`traceloop.initialize(...)` in TS) registers its provider globally.

*Python.* OTel enforces a set-once latch on the global provider, so
if kubit-otel registers first, Traceloop's registration silently
becomes a no-op and Traceloop spans are lost. The Python §3
snippet uses `attach()` and must run **after** `Traceloop.init(...)`
— it raises loudly if invoked too early rather than silently
dropping spans.

*TypeScript (OTel JS SDK v2).* There is no post-hoc attach on Node —
v2 removed `addSpanProcessor` and `@kubit-ai/otel` does not export
`attach()`. Kubit must be added as `new KubitSpanProcessor({ apiKey })`
to a `NodeSDK` / `NodeTracerProvider` the repo constructs alongside
`traceloop.initialize(...)`. If the repo only calls
`traceloop.initialize(...)` without standing up its own `NodeSDK`,
there is no v2-compatible merge target — see §3 TS for the
reference snippet the skill prints in that case.

## 3. Bootstrap snippet

Reference Kubit wiring — the *minimum* code that must end up in the
program (import + processor/`configure` call, plus any assertions
below). When merging into an existing wiring site (see §3a), adapt
placement and syntactic style to the surrounding file. The standalone
snippet is used verbatim only when no site is found.

**Gotcha — undocumented workaround.** Traceloop does not expose its
`TracerProvider` directly, and `Traceloop.init(exporter=...)` only
accepts a single custom exporter (replacing the Traceloop default).
Attaching Kubit via `trace.get_tracer_provider().add_span_processor(...)`
after `Traceloop.init(...)` works today because Traceloop registers
its provider globally, but it is not a vendor-supported pattern —
tracked upstream as
[traceloop/openllmetry#3478](https://github.com/traceloop/openllmetry/issues/3478).
If a future Traceloop release switches to a private provider this
will silently stop picking up spans; re-run `/kubit-integrate` to
refresh the wiring in that case.

### Python — `kubit_instrumentation.py`

```python
# Generated by /kubit-integrate for openllmetry on <YYYY-MM-DD>.
# Must be imported AFTER Traceloop.init().
# Requires KUBIT_EXPORT_API_KEY and KUBIT_EXPORT_ENDPOINT env vars.
import os
from kubit_otel import attach

attach(
    api_key=os.environ["KUBIT_EXPORT_API_KEY"],
    token_endpoint=os.environ["KUBIT_EXPORT_ENDPOINT"],
)
```

### TypeScript — merge into the Traceloop / OTel provider construction site

OTel JS SDK v2 requires processors at provider construction time;
`@kubit-ai/otel` does not export `attach()`. Repos that stand up
their own `NodeSDK` / `NodeTracerProvider` alongside
`traceloop.initialize(...)` add `KubitSpanProcessor` to that site's
`spanProcessors: [...]`:

```typescript
import { NodeSDK } from "@opentelemetry/sdk-node";
import { KubitSpanProcessor } from "@kubit-ai/otel";
// …existing Traceloop imports stay in place…

const sdk = new NodeSDK({
  spanProcessors: [
    // …existing Traceloop processor stays first…,
    new KubitSpanProcessor({ apiKey: process.env.KUBIT_EXPORT_API_KEY! }),
  ],
});
sdk.start();
```

If the repo only calls `traceloop.initialize(...)` and never
constructs a `NodeSDK` / `NodeTracerProvider`, there is no
v2-compatible merge target — the skill prints this snippet as a
reference and instructs the user to stand up a `NodeSDK` that
carries both processors. The skill must not emit a standalone file
that calls `configure({ ... })`, because that would register a
second, parallel `NodeTracerProvider` and clobber Traceloop's
registration.

## 3a. Integration-site signals

Grep targets for an existing wiring site to merge Kubit into. If
exactly one file in the repo matches, merge §3 wiring into that file;
otherwise fall back to the standalone bootstrap file per SKILL.md
step 9.

Python:

- A call to `Traceloop.init(` or `traceloop.sdk.Traceloop.init(`. Call
  `attach({...})` from `kubit_otel` on the next line — it resolves the
  provider Traceloop registered globally and raises if `init()` has
  not run yet.

TypeScript:

- A `new NodeSDK({ spanProcessors: [...] })` or
  `new NodeTracerProvider({ spanProcessors: [...] })` construction
  in a module that also imports from `@traceloop/node-server-sdk` —
  add
  `new KubitSpanProcessor({ apiKey: process.env.KUBIT_EXPORT_API_KEY! })`
  to the same `spanProcessors` array.
- A call to `traceloop.initialize(` from `@traceloop/node-server-sdk`
  **without** a local `NodeSDK` / `NodeTracerProvider` construction
  — no v2-compatible merge site; print the §3 merge snippet as a
  reference and ask the user to stand up a `NodeSDK` that hosts
  both processors.

If multiple files match, ask the user which one the agent's traces
flow through.

## 4. Wire-in instruction

Python: `{{KUBIT_IMPORT_STATEMENT}}` after `Traceloop.init(...)`.
TypeScript: the merge edit lands directly in the `NodeSDK` /
`NodeTracerProvider` construction site — no separate bootstrap file,

Required deps:

- Python: `pip install kubit-otel`
- TypeScript: `npm install @kubit-ai/otel` (requires
  `@opentelemetry/sdk-trace-base >= 2.0.0` as a peer — pin

  `resources` to the same `^2.x` major the project already uses).
