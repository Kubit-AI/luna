## 2. Minimum-change tier

`bootstrap-file`

Phoenix owns the `TracerProvider` — `phoenix.otel.register(...)`
constructs it and registers it globally.

*Python.* OTel enforces a set-once latch on the global provider, so
if kubit-otel registers first, Phoenix's registration silently
becomes a no-op and Phoenix spans are lost. The Python standalone
snippet in §3 therefore uses `attach()` and must run **after**
`phoenix.otel.register(...)` — it raises loudly if invoked too
early rather than silently dropping spans.

*TypeScript (OTel JS SDK v2).* There is no post-hoc attach on Node:
v2 removed `addSpanProcessor` and `@kubit-ai/otel` does not export
`attach()`. Kubit must be added as `new KubitSpanProcessor({ apiKey })`
in the `spanProcessors: [...]` array at the
`NodeTracerProvider` / `NodeSDK` construction site that Phoenix's
`register({...})` helper wraps (or the site the repo hand-builds).

## 3. Bootstrap snippet

Reference Kubit wiring — the *minimum* code that must end up in the
program (import + processor/`configure` call, plus any assertions
below). When merging into an existing wiring site (see §3a), adapt
placement and syntactic style to the surrounding file. The standalone
snippet is used verbatim only when no site is found.

### Python — `kubit_instrumentation.py`

```python
# Generated by /kubit-integrate for openinference on <YYYY-MM-DD>.
# Must be imported AFTER phoenix.otel.register().
# Requires KUBIT_EXPORT_API_KEY and KUBIT_EXPORT_ENDPOINT env vars.
import os
from kubit_otel import attach

attach(
    api_key=os.environ["KUBIT_EXPORT_API_KEY"],
    token_endpoint=os.environ["KUBIT_EXPORT_ENDPOINT"],
)
```

### TypeScript — merge into the Phoenix / OTel provider construction site

OTel JS SDK v2 requires processors at provider construction time;
`@kubit-ai/otel` does not export `attach()`. There is no standalone
Node file — the skill edits the Phoenix or hand-built provider
construction in place, adding `KubitSpanProcessor` to
`spanProcessors: [...]`:

```typescript
import { NodeSDK } from "@opentelemetry/sdk-node";
import { KubitSpanProcessor } from "@kubit-ai/otel";
// …existing Phoenix / OpenInference imports stay in place…

const sdk = new NodeSDK({
  spanProcessors: [
    // …existing OpenInference / Phoenix processor stays first…,
    new KubitSpanProcessor({ apiKey: process.env.KUBIT_EXPORT_API_KEY! }),
  ],
});
sdk.start();
```

If the repo calls `register({...})` from `@arizeai/phoenix-otel`
without also constructing a `NodeSDK` / `NodeTracerProvider`, there
is no v2-compatible merge target — the skill prints this snippet as
a reference and instructs the user to stand up a single `NodeSDK`
that carries both processors.

## 3a. Integration-site signals

Grep targets for an existing wiring site to merge Kubit into. If
exactly one file in the repo matches, merge §3 wiring into that file;
otherwise fall back to the standalone bootstrap file per SKILL.md
step 9.

Python:

- A call to `phoenix.otel.register(` — the returned provider is the
  target for `provider.add_span_processor(KubitSpanProcessor(...))`.
  If the call is shadowed in a named variable
  (e.g. `provider = register(...)`), reuse that variable. When no
  named variable is in scope, call `attach(...)` from `kubit_otel`
  on the next line instead — it resolves the provider via the global
  API and raises if `register()` has not run yet.
- A module that already imports `phoenix.otel` and constructs its own
  `TracerProvider` (rare) — merge onto that provider.

TypeScript:

- A `new NodeSDK({ spanProcessors: [...] })` or
  `new NodeTracerProvider({ spanProcessors: [...] })` /
  `new BasicTracerProvider({ spanProcessors: [...] })` construction
  in a module that also imports from `@arizeai/phoenix-otel` or
  `@arizeai/openinference-*` — add
  `new KubitSpanProcessor({ apiKey: process.env.KUBIT_EXPORT_API_KEY! })`
  to the same `spanProcessors` array.
- A call to `register({...})` from `@arizeai/phoenix-otel`
  **without** a local `NodeSDK` / `NodeTracerProvider` construction
  — no v2-compatible merge site; print the §3 merge snippet as a
  reference and ask the user to stand up a `NodeSDK` that hosts
  both processors.

If multiple files match, ask the user which one the agent's traces
flow through.
